## 1. 这份文档的目标

这份文档用于给当前“培养阶段”做一个阶段性收口。

它回答四个问题：

1. 现在培养这条线已经做到了什么程度
2. 当前系统已经具备哪些稳定能力
3. 当前边界和风险点在哪里
4. 为什么下一阶段应该把重点切到“客户端 + 交流层”

## 2. 当前阶段已经完成的事情

### 2.1 输入侧已经从单纯 RawDocument 走到统一证据层

当前仓库已经不再只是“抓一堆文本直接喂训练”。

已经落地的输入结构包括：

1. `RawDocument`
2. `TargetManifest`
3. `EvidenceItem`
4. `EvidenceBatch`

已经接入或预留的输入来源包括：

1. Twitter/X 推文
2. 文章
3. 本地聊天记录
4. 视频/音频 transcript

这意味着培养阶段的输入底座已经不是一次性的 prompt 拼接，而是可追踪、可分段、可归因的证据对象。

### 2.2 路由层已经从单轨升级到双轨

当前稳定存在两套路由：

1. `legacy`
2. `v2`

其中：

1. `legacy` 仍是默认安全主线
2. `v2` 已经具备 `Soul / Memory / Discard` 分流能力
3. `v2` 已经有真实实验资产支撑，不再只是概念

### 2.3 训练阶段已经形成闭环

当前训练链路已经完整跑通：

`输入 -> 证据整理 -> 路由 -> Soul 提炼 -> Memory 构建 -> TrainingLoop -> 评估 -> 记忆治理 -> 报告`

已经稳定存在的核心角色：

1. `PersonaAgent`
2. `TrainerAgent`
3. `EvaluatorAgent`
4. `DirectorAgent`

已经稳定存在的核心产物：

1. `persona.json`
2. `soul.yaml`
3. `training-report.json`
4. `training-seed.json`
5. shard / global merge 中间资产

### 2.4 大语料路径已经具备工程底座

当前“大语料”虽然还没有到最终生产形态，但底座已经成立：

1. `Corpus Snapshot`
2. `Shard Plan`
3. `Shard Distillation`
4. `Global Merge`
5. `PK aggregate`

这意味着系统已经具备：

1. 分片处理
2. 分片观测
3. 全局合并
4. 异常 run 隔离
5. `raw mean` / `clean mean` 双口径报告

## 3. 当前已经验证过的能力边界

### 3.1 已验证成立的能力

当前可以认为已经被验证成立的能力：

1. 公开文本语料的人格蒸馏
2. `legacy` 与 `v2` 的真实对照实验
3. provider 抖动下的异常 run 隔离
4. `1000+` 到 `4000+` 级别公开推文的扩容、蒸馏与 PK
5. `chat-once` / `chat` 形式的对话入口
6. 从培养产物导出到 OpenClaw / LobeChat 配置

### 3.2 当前的“读能力”

如果从“交流前要先把读能力培养好”的角度看，当前系统已经有以下读取能力：

1. 读取目标人物的公开语料
2. 读取并结构化人格资产 `Soul`
3. 读取长期记忆 `Memory`
4. 读取当前会话历史
5. 读取训练阶段积累出的 `training seed`
6. 读取聊天/视频的统一证据对象

也就是说，系统已经能“读”：

1. 目标人物过去说过什么
2. 目标人物稳定的表达方式和价值倾向
3. 目标人物在特定话题上的长期记忆
4. 当前用户正在和它聊什么

### 3.3 当前的“写能力”

当前系统也已经有基础写能力，但还主要服务于培养，不是服务于产品化交流：

1. 写 `Soul`
2. 写 `Memory`
3. 写训练报告
4. 写 shard / merge 中间资产
5. 写导出配置
6. 写单轮聊天回复

但这里有一个关键区分：

当前“写”的主能力是：

1. 往培养资产里写
2. 往实验报告里写
3. 往导出配置里写

而不是：

1. 在真实用户会话里做稳定的会话级写回
2. 在客户端内形成持续的交流状态管理

这也是为什么现在适合切到客户端阶段。

## 4. 当前阶段的核心结论

### 4.1 培养主线已经不是阻塞项

当前的培养主线已经具备以下特征：

1. 有稳定默认路径
2. 有灰度实验路径
3. 有真实账号对照
4. 有异常隔离能力
5. 有中间资产可回放

所以现在的主要矛盾已经不再是“能不能培养出来”，而是：

1. 培养好的东西怎么稳定地拿来交流
2. 交流过程中的新信息怎么写回
3. 写回哪些层，写到哪一层，怎么避免污染 Soul

### 4.2 当前不应该再把所有注意力都放在继续扩容

扩容仍然重要，但它已经不是唯一主线。

当前更合适的节奏是：

1. 保留培养主线持续可跑
2. 保留大语料实验能力
3. 把主开发重心切到客户端与交流层

原因很简单：

1. 没有客户端，培养成果很难形成产品闭环
2. 没有交流层，系统无法验证“培养结果是否真的可用”
3. 没有写回策略，客户端越强，污染人格资产的风险越高

## 5. 当前阶段不应该做什么

当前不建议：

1. 继续把所有资源压在更大规模公开推文实验上
2. 在还没有客户端交流层的前提下，直接做复杂在线写回
3. 让会话过程直接写 `Soul`
4. 把 `signals` 提前升成默认路径
5. 在没有会话治理的前提下，把“培养完成”误判为“产品完成”

## 6. 下一阶段应该切什么

下一阶段建议从“培养系统”切到“交流系统”。

也就是从：

`如何培养`

转到：

`培养好的 persona 如何被使用、如何持续交流、如何安全写回`

下一阶段优先级建议：

1. 客户端 MVP
2. 交流层读写架构
3. 会话写回治理
4. 再继续扩大多模态和更大语料

## 7. 一句话总结

如果用一句话总结当前阶段：

> 培养这条线已经形成了可用、可实验、可扩展的 V1 闭环，下一阶段最应该补的不是“再多训练一点”，而是“让培养好的 persona 真正开始稳定交流”。
